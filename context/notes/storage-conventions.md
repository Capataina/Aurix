# Storage Conventions

## Current Understanding

The [storage](../systems/storage.md) layer enforces a small set of conventions across every per-table CRUD module. Violating any one of them produces a class of subtle bug — silent zero-row queries, lost idempotency, or false synthetic-vs-live overlap — that the existing test suite cannot fully catch. Capture them once, here, so future code changes preserve the contract.

## The Conventions

### 1. `INSERT OR IGNORE` keyed on a domain-natural unique constraint

Every batch-insert path uses `INSERT OR IGNORE` rather than `INSERT` or `INSERT OR REPLACE`. The unique constraint is a domain-natural composite, not a synthetic auto-increment id:

| Table | Unique constraint |
|---|---|
| `swap_events` | `(pool_address, block_number, log_index)` — chain-globally unique |
| `pool_events` | `(pool_address, block_number, log_index)` — same |
| `position_runs` | `config_hash` — deterministic SHA over the `PositionConfig` |
| `headline_runs` | `config_hash` — deterministic SHA over the `HeadlineConfig` |
| `strategy_results` | `(grid_id, cell_index)` — grid-internal natural key |
| `benchmark_series` | `(series_key, date)` — note: this one uses `INSERT OR REPLACE` because providers revise their series; see exception below |

**Exception — `benchmark_series` uses `INSERT OR REPLACE`** because external providers (FRED especially) revise older series prints. Replacing on collision keeps the local cache aligned with provider truth.

**Why this matters.** Re-running ingest, backtest, or grid execution against identical inputs is structurally a no-op. The frontend's auto-run pipeline depends on this — every page mount fires the pipeline, and idempotency means cold-start performs no DB churn beyond reading the cached results.

### 2. Lowercase pool addresses on **both** insert and query paths

`storage::swaps::insert_swap_events_batch` and `storage::pool_events::insert_pool_events_batch` lowercase `pool_address` before the SQL `params!`. Every read path that compares pool addresses also lowercases.

**Why this matters.** Live Alchemy logs arrive with lowercase addresses by chain convention. User-supplied addresses arrive EIP-55-checksummed (mixed case). Storing them mixed-case and querying lowercase produces zero-row results — exactly the bug commit `53f99eb` fixed: the synthetic mock used the user's mixed-case input, then the backtest queried with the lowercased version, hit zero rows, and the LP page silently rendered as empty.

The storage layer is the canonical normalisation point. Callers can pass any case; the DB always holds lowercase.

### 3. Synthetic-vs-live separation via `SYNTHETIC_TX_HASH`

Every synthetic swap row (generated by `commands::lp::synthetic_mock`) carries `transaction_hash = SYNTHETIC_TX_HASH = "0x...deadbeef"`. Real Ethereum tx hashes are uniformly distributed 256-bit; collision probability with the all-zero-prefix sentinel is ~2⁻²²⁴, effectively zero.

`Storage::delete_synthetic_swaps_in_range(pool, from, to)` purges only synthetic rows in the range. Live ingestion never touches rows with this hash.

**Why this matters.** Without the sentinel + the targeted purge, the synthetic generator could not be re-run after parameter changes (e.g. tick anchor, active liquidity calibration) — `INSERT OR IGNORE` would silently keep the stale rows. The wipe-then-insert pattern preserves the idempotency contract for live data while letting synthetic data be regenerated cleanly.

### 4. TEXT decimal strings for big-integer columns

`amount0`, `amount1`, `sqrt_price_x96`, `liquidity`, and `tick` are stored as TEXT decimal strings (except `tick`, which fits in `i32` and is stored as INTEGER). Callers parse to `BigInt` / `BigUint` / `u128` on read.

**Why this matters.** SQLite's INTEGER type is 64-bit signed; sqrtPriceX96 is uint160, liquidity is uint128, amount0/amount1 are int256. Storing these as INTEGER would lose precision for values above 2⁶³. TEXT decimal preserves the full range. The wire-side `f64` representation in `EquityCurvePoint` (USD values) is acceptable because monetary precision is bounded; the storage representation must support the full uint160/int256 domain.

The trade-off: every read pays a `BigUint::parse_bytes` cost. See [audit findings](../plans/code-health-audit/backtest.md) §"Pre-parse swap rows once" for the per-loop perf implication and the recommended hoist.

## Guiding Principles

- New CRUD modules follow the same conventions: idempotent insert + lowercase-address (where applicable) + sentinel-tx-hash separation (where synthetic data can exist) + TEXT decimal for >64-bit integers.
- Test the idempotency contract: every batch-insert function gets a `*_is_idempotent` test that inserts the same batch twice and asserts the second insert returns 0 (per `storage::swaps::tests::batch_insert_is_idempotent`, `storage::pool_events::tests::round_trip_mints_and_burns`, etc.).
- Test the address-case round-trip: at least one test per address-bearing table inserts with mixed-case input and queries with lowercase, asserting non-zero result.

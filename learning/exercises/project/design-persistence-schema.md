# Exercise: Design the Persistence Schema

## Goal

Design the SQLite schema for Aurix's persistence layer (Vector A's M2.0). This unblocks Tab 1's historical features, all of Vectors A and C, and most of Tabs 4 and 5. The schema design will outlast the initial use case — get it right.

This is a design exercise. Don't implement; design carefully and write the schema as DDL.

## Estimated Time

60-90 minutes.

## Setup

Read:
- `context/plans/vector-a-v3-lp-backtester.md` (M2.0 specification)
- `context/plans/vector-c-ml-arbitrage-survival.md` (M-ML.1 — same persistence layer, different consumers)
- `src-tauri/src/market/types.rs` (the current MarketOverview shape)

The schema must support:

1. **Tab 1 historical mode** — replay any past window of MarketOverviews for the historical chart
2. **Vector A backtesting** — store V3 swap events for tick-by-tick replay
3. **Vector C training** — store per-tick snapshots for ML feature engineering
4. **Tab 4 gas history** — store per-tick gas readings for hour-of-day analysis
5. **Tab 5 multi-token expansion** (future) — should not require a migration to add WBTC/ETH

## Tasks

### Part 1 — Identify the entities

What "things" need to be persisted? List them. For each, name the granularity (per-tick, per-block, per-event, per-position).

Examples to consider:
- Market snapshots (one per tick, current)
- Per-venue prices (one per venue per tick — could be a child table)
- Swap events (one per Uniswap swap that happens — for Vector A)
- Gas history (could be derived from snapshots, or its own table)
- ML predictions (one per tick once Vector C ships)
- Backtest runs (one per backtest user runs, with parameters)
- LP simulation results (output of a backtest run)

### Part 2 — Decide table structure

For each entity, write the SQLite DDL. Decide:
- [ ] Primary key (auto-increment? compound? timestamp?)
- [ ] Foreign keys (which entities relate to which)
- [ ] Indexes (which queries are common — and what indexes do they need?)
- [ ] Data types (REAL vs INTEGER vs TEXT for prices, addresses, IDs)

### Part 3 — Critical schema decisions

Walk through these specific decisions:

#### Decision 1 — One-table-per-venue or one-table-with-venue-column?

Two designs:

```sql
-- Option A: per-venue tables
CREATE TABLE v3_5bps_snapshots (timestamp INTEGER, price REAL, ...);
CREATE TABLE v3_30bps_snapshots (timestamp INTEGER, price REAL, ...);
CREATE TABLE v2_snapshots (timestamp INTEGER, price REAL, ...);
CREATE TABLE sushi_snapshots (timestamp INTEGER, price REAL, ...);

-- Option B: one table with a venue column
CREATE TABLE venue_snapshots (
  timestamp INTEGER,
  venue TEXT,
  price REAL,
  ...
);
```

Which would you choose? Why? What does the choice unlock or block?

#### Decision 2 — Store prices as REAL or as TEXT?

`REAL` is f64 (15-16 significant digits). Prices like $3,047.234 are well within precision.

`TEXT` would let you store the original BigUint value as a decimal string with arbitrary precision (matching the on-chain reality exactly).

Trade-offs?

#### Decision 3 — Schema versioning

How will you handle schema changes when Vector A or B is added? Three options:

- a) `schema_version` table with current version number
- b) Migrations folder (forward-only, applied in order, recorded in `migrations_applied` table)
- c) Drop and recreate on every version change (data is local, can be regenerated by re-fetching)

Which fits Aurix best? Why?

#### Decision 4 — Storage budget

Aurix polls at 1 Hz. Each tick is ~5 venue snapshots + 1 gas reading.

- 1 day at 1 Hz = 86,400 snapshots = ~5 MB at 50 bytes/row average
- 30 days = ~150 MB
- 365 days = ~1.8 GB

Is this acceptable? At what point do you need a retention policy? What kind?

#### Decision 5 — Concurrency and WAL

Aurix's frontend reads while the backend writes. SQLite supports several concurrency modes:

- Default journal mode: `DELETE` (writers exclude readers)
- WAL mode (`PRAGMA journal_mode = WAL`): writers and readers can coexist

Which would you use? Why?

(The Image Browser project already uses WAL with split read/write connections — check `src-tauri/Cargo.toml` of that project for the pattern if you have access.)

### Part 4 — Write the schema

Write the complete DDL as you'd ship it to M2.0. Include:

- All `CREATE TABLE` statements
- All `CREATE INDEX` statements
- Any `PRAGMA` settings (e.g. `journal_mode = WAL`)
- Comments explaining non-obvious choices

### Part 5 — Validate against query patterns

For each consumer use case below, write the SQL query that would satisfy it. If your schema doesn't support a query efficiently (would require full table scan), reconsider the schema.

- [ ] Tab 1: get the last 1000 MarketOverviews ordered by timestamp DESC
- [ ] Vector A: get all V3 5bps swap events between block A and block B
- [ ] Vector C training: get all per-tick snapshots for the last 7 days, joined with their 30-second-forward labels
- [ ] Tab 4: aggregate gas prices by hour-of-day over the last 30 days
- [ ] Tab 5 (future): get USDC/WETH and USDC/WBTC prices at the same timestamps for correlation analysis

If any of these queries are awkward, your schema needs adjustment.

## Hints

### Hint 1

The big design tension is between "tab-scoped tables" (one table per tab's needs, simple but redundant) and "domain-modeled tables" (entities like 'snapshot' shared across tabs, normalised but more JOINs).

For Aurix, lean toward domain-modelled. The five tabs share many concepts (snapshots, prices, venues, gas) — duplicating them per-tab creates maintenance burden and prevents cross-tab analysis.

### Hint 2

A good starting structure (sketch — not the full answer):

```sql
-- Tracks every poll tick
CREATE TABLE market_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp_ms INTEGER NOT NULL,
  chain TEXT NOT NULL,
  pair_label TEXT NOT NULL,
  gas_price_gwei REAL NOT NULL
);
CREATE INDEX idx_snapshots_time ON market_snapshots(timestamp_ms);

-- Tracks per-venue price for each snapshot
CREATE TABLE venue_prices (
  snapshot_id INTEGER NOT NULL REFERENCES market_snapshots(id),
  venue TEXT NOT NULL,
  price_usd REAL NOT NULL,
  fee_tier_bps INTEGER NOT NULL,
  pool_address TEXT NOT NULL,
  PRIMARY KEY (snapshot_id, venue)
);

-- Tracks Uniswap V3 swap events from on-chain (for Vector A)
CREATE TABLE v3_swaps (
  ...
);

-- Tracks user-initiated backtest runs
CREATE TABLE backtest_runs (
  ...
);
```

Continue this design through the missing entities. Add indexes thoughtfully (every query needs to be index-supported).

### Hint 3

For schema versioning, option (b) "migrations folder" is the most flexible and the standard pattern. A bare-bones implementation:

```rust
// src-tauri/src/storage/migrations.rs
const MIGRATIONS: &[&str] = &[
    include_str!("migrations/001_initial.sql"),
    include_str!("migrations/002_add_v3_swaps.sql"),
    // ...
];

// On startup:
//   read schema_version from db (or 0 if uninitialised)
//   apply MIGRATIONS[schema_version..] in sequence
//   update schema_version to MIGRATIONS.len()
```

This lets you ship schema changes without losing user data, and each migration is reviewable independently.

## Expected Behaviour / Self-Check

A good schema:

- Supports all 5 consumer queries in Part 5 with index-backed performance (no table scans on production-scale data)
- Doesn't require a migration to add a new venue (if you hard-coded `v3_5bps_snapshots`, you'd need a new table for every new venue)
- Is honest about precision (REAL is fine for display; if Vector A needs exact integer math, capture raw `sqrtPriceX96` as TEXT alongside the f64 price)
- Has explicit versioning so future migrations are clean
- Documents the WAL mode choice and the read/write connection split

If your design has any "we'll figure that out later" placeholders for the consumer queries, those are red flags. Resolve them before declaring the design done.

## What You Should Take Away

- Schema design is forward-looking — the wrong schema costs migrations later
- One table with a venue column > one table per venue (almost always)
- Indexes are query-driven; design them after the query patterns are known
- WAL mode is the right choice for desktop apps with concurrent reads/writes
- Migrations should be forward-only and code-reviewed
